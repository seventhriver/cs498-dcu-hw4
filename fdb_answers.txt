Q1 (3 pts) — FoundationDB separates its architecture into a Transaction Management System (TMS) and a Storage System (SS). Why does FDB decouple these two components, and what does each layer handle?

- This allows the two components to act separately, and to be managed separately. The Storage System is removed from the direct process of commits, and the write and concurrency control processes can act independently of read requests, which means a) load on one end only directly affects that end, b) the two systems can be scaled independently of each other, and c) recovery is easier because the systems only need to stabilize/manage their own concerns.
- The transaction management system handles the write process, concurrency concerns and conflict checks, and logging commits to be stored. The storage system handles long-term storing the database state (and committed transactions) on disk, as well as serving read requests (read requests also contact the Sequencer, which is technically part of the TMS, for a read version).

Q2 (3 pts) — During a write transaction in FDB, which component is responsible for detecting read-write conflicts between concurrent transactions, and at what point in the commit process does this check occur?

- The Resolver is responsible for detecting conflicts on write requests, and it does so after it has received the transaction to be committed but before it has actually been committed (logged). It checks the request's read version, and if any other transaction has modified the attempted commit's read set since that version, the commit is aborted and the client has to get a new read and retry.

Q3 (3 pts) — FDB's Sequencer assigns both a read version and a commit version to each transaction. What is the purpose of the read version, and how does it relate to the consistency guarantees FDB provides?

- As described in question 2, the read version allows the resolver to check if the transaction has a write conflict. If any other transaction *since the attempted transaction's read version* has modified any value in the attempted transaction's read set, the commit process is aborted and the client has to retry. This makes sure no data is overwritten / avoids write conflicts, and is what gives FDB its serializability. Also, the transaction's view of the database is frozen at the read version, which makes sense (that is the version of the database the transaction "got" on read).

Q4 (3 pts) — FDB supports snapshot reads. How do snapshot reads differ from standard reads with respect to conflict detection, and in what situation would you choose one over the other?

- Snapshot reads work similarly in the way that they grab a view of the database at the start of the transaction, however, they don't generate/add to the transaction's read set. This means they don't track conflicts and will commit even if some of the data they read has been changed since they "saw" it. This doesn't provide strict guarantees that the data used in the transaction is up-to-date, but can be better for situations where this is acceptable (such as aggregations on frequently changing data or streams, where approximate freshness is ok) since it's guaranteed to commit on the first try (unless something else fails, of course).

Q5 (3 pts) — FDB claims that its recovery is "cheap" and does not require replaying a redo log in the critical path. Explain what makes this possible, referencing how StorageServers and LogServers interact during normal operation.

- In a traditional database, the current database state is stored in memory, and snapshots are taken to a disk storage system. So, if the state memory fails, you'd have to replay all transactions from the latest snapshot. However, for FDB, the storage system has a near-current version of the database that asynchronously pulls commits from the (durable!) logging system and applies them -- if a failure occurs in FDB, you just need to start up a new Transaction Management System (which can almost immediately begin accepting new writes), separate off the block of committed/logged but un-stored transactions in the old log servers, and allow the storage servers to continue pulling from them and then eventually transition to the new LogServers/TMS.
